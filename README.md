# Первая практическая работа по работе с Git
Вам необходимо сделать форк (```fork```) данного репозитория на свой аккаунт GitHub и клонировать (```clone```) его на свой локальный компьютер. Далее, перед началом работы, Вам необходимо создать новую ветку (```branch```), назвав её фамилией(ями) выполняющих работу студентов (например, ```bogdanov_zanin```), в случае наличия однофамильцев в группе, укажите и инициалы. 

После создания ветки, скачайте из релизов (```Releases```) программу, выдающую номера задач по номеру академической группы и Вашим фамилиям.
> ***Обратите внимание, некорректный ввод в программу, а в следствии и неправильные номера задач, исключительно на Вашей ответственности. Решения не Ваших задач проверяться не будут.***

После получения задач, найдите их ниже и приступайте к их выполнению в ранее клонированном репозитории на любом удобном Вам языке программирования, создавая коммиты (```commit```) по смысловым шагам (как минимум после решения каждой из задач). Каждая задача должна иметь полное решение, продумайте логику и протестируйте работоспособность, код должен быть чистым и понятным, также не забудьте про наличие комментариев, где это необходимо.

Требуемая структура репозитория:
```
<номер группы>/        ← глобальная папка с номером группы
  <имена студентов>/   ← как и у ветки (например, bogdanov_zanin)
    task01/
      solution.<ext>
      README.md        ← описание решения конкретной задачи
      tests/
    task07/
      solution.<ext>
      README.md
    task12/
      solution.<ext>
      README.md
```
Пример README.md для конкретной задачи:
```
# Задача 07 — Симулятор очереди

## Участники группы
Богданов М.М.
Занин П.Р.

## Описание решения
Использована событийная модель...
Очередь реализована через...

## Как запустить
python solution.py input.txt
```

После завершения работы необходимо запушить ```push``` изменения в свой форк ```fork``` и сделать ```Pull Request``` в данный репозиторий.
> ***Обратите внимание, только после принятия реквеста будет считаться, что Вы сдали работу. Также очень важно, чтобы структура вашего репозитория соответствовала ранее описанной, если это не так, ```Pull Request``` может быть отклонён.***

Если по какой-то причине я не приму ```Pull Request```, причины этого я опишу в комментариях к нему.
## Задачи
<details>
<summary>1) Проверка корректности скобок</summary>

### Цель: проверить, корректно ли закрыты скобки в строке
### Требования:
* Поддерживать (), [], {}
* Игнорировать другие символы
* Вернуть ```YES``` / ```NO``` или ```true``` / ```false```
</details>

<details>
<summary>2) Подсчёт слов</summary>

### Цель: подсчитать строки, слова и найти самое длинное слово в тексте
### Требования:
* Считать файл
* Отделять слова по пробельным символам и пунктуации
* Вернуть количество строк, слов, самое длинное слово (и его длину)
</details>

<details>
<summary>3) Переворот слов в тексте</summary>

### Цель: перевернуть порядок слов в тексте
### Требования:
* Считать файл
* Сохранять слова как блоки (не переворачивать буквы)
* Удалить лишние пробелы в начале/конце и между словами оставить один пробел
</details>

<details>
<summary>4) Удаление дубликатов, сохраняя порядок</summary>

### Цель: удалить повторяющиеся элементы, сохранив исходный порядок
### Требования:
* Считать файл (в нём записаны слова или числа через пробел)
</details>

<details>
<summary>5) Римские числа (двунаправленный конвертер)</summary>

### Цель: преобразование между римскими и арабскими числами
### Требования:
* Поддержка значений от 1 до 3999
* Валидация римской строки (правила составления)
### Примеры:
```
1987 → MCMLXXXVII
IV → 4
```
</details>

<details>
<summary>6) Реализация сортировки (без встроенных)</summary>

### Цель: реализовать один из алгоритмов сортировки (merge/quick/bubble)
### Требования:
* Нельзя использовать встроенные функции и методы сортировки
* Оцените сложность алгоритма в ```READMEmd```
</details>

<details>
<summary>7) Подсчёт частоты символов</summary>

### Цель: вывести частоту каждого символа в файле
### Требования:
* Игнорировать/не игнорировать регистр - опция
* Сортировка вывода по убыванию частоты
</details>

<details>
<summary>8) Генератор пароля</summary>

### Цель: сгенерировать случайный пароль заданной длины с наборами символов
### Требования:
* Опции: включать/исключать цифры, прописные/строчные буквы, спецсимволы
* Обязательно минимум по одному символу из включённых наборов (если длина позволяет)
</details>

<details>
<summary>9) Сжатие строки (Run-Length Encoding)</summary>

### Цель: реализовать RLE-сжатие
### Пример:
```
AAABBBCCDAA → A3B3C2D1A2
```
### Требования:
* Реализовать также декодирование
* Поддержка любых символов
</details>

<details>
<summary>10) Проверка IP-адреса</summary>

### Цель: определить, является ли строка корректным IPv4 или IPv6 адресом
### Вывод: ```IPv4```, ```IPv6``` или ```INVALID```
### Бонус: нормализовать IPv6 (убрать лишние нули)
</details>

<details>
<summary>11) Проверка шахматного хода</summary>

### Цель: проверить, может ли фигура сделать ход
### Ввод: файл где записано состояние шахматной доски (8 строк по 8 символов)
### Обозначения:
* Заглавные буквы - белые, строчные - черные
* ```K``` - Король (King)
* ```Q``` - Ферзь (Queen)
* ```R``` - Ладья (Rook)
* ```B``` - Слон (Bishop)
* ```N``` - Конь (Knight, так как K занято королем)
* ```P``` - Пешка (Pawn)
### Требования:
* Учесть столкновение фигур
### Вывод: ```VALID``` / ```INVALID```
</details>

<details>
<summary>12) Разбор HTML-тегов</summary>

### Цель: проверить, корректно ли вложены HTML-теги в документе
### Требования:
* Считать файл
* Игнорировать все пробелы, переносы строк и табуляции
### Критерии валидности:
* Теги должны быть закрыты в обратном порядке открытия
```
<div><p></p></div>      → VALID
<div><p></div></p>      → INVALID
```
* Каждый открывающий тег должен иметь закрывающий (если это не самозакрывающийся тег)
```
<div><span></span></div>  → VALID
<div><span></div>         → INVALID
```
* Следующие теги не требуют закрывающего: ```<br>```, ```<hr>```, ```<img>```, ```<input>```, ```<meta>```, ```<link>```
* Также вариант с / допустим
```
<img src="ajpg"/>
```
* Атрибуты должны игнорироваться Для проверки важны только имена тегов
```
<div class="box" id="main"></div>
```
* Регистр не важен: ```<DIV>``` и ```<div>``` - один и тот же тег
### Упрощения:
* комментарии ```<!-- -->``` можно игнорировать или считать обычным текстом
* ```<!DOCTYPE>``` можно игнорировать
* ```<script>``` и ```<style>``` не требуют спецобработки
* сущности ```&nbsp;``` игнорируются
* не нужно проверять допустимость структуры HTML по стандарту (например, ```<li>``` вне ```<ul>```)
### Вывод: ```VALID``` / ```INVALID```
</details>
